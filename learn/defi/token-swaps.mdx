## How DEX swaps work

A **decentralized exchange (DEX)** lets you swap one token for another without a centralized order book. The most common model is an **automated market maker (AMM)** — a smart contract that holds reserves of two tokens and prices them using a mathematical formula.

The classic formula (Uniswap v2 style):

```
x × y = k
```

Where `x` and `y` are the pool's token balances and `k` is a constant. Buying token Y removes Y from the pool and adds X, pushing the price of Y up relative to X. The larger your trade relative to the pool size, the more price impact.

## Uniswap v3: concentrated liquidity

Uniswap v3 extends the AMM model with **concentrated liquidity**: liquidity providers choose a price range, and their capital is only active when the current price falls within that range. This produces tighter spreads and better prices for traders.

For agents executing swaps, the key function is `exactInputSingle` on the `SwapRouter02` contract:

```solidity
function exactInputSingle(ExactInputSingleParams calldata params)
    external
    returns (uint256 amountOut);

struct ExactInputSingleParams {
    address tokenIn;
    address tokenOut;
    uint24 fee;               // pool fee tier: 100, 500, 3000, or 10000
    address recipient;
    uint256 amountIn;
    uint256 amountOutMinimum; // slippage protection
    uint160 sqrtPriceLimitX96;
}
```

Fee tiers map to: `100` = 0.01%, `500` = 0.05%, `3000` = 0.3%, `10000` = 1%. Most stablecoin pairs are on `500`; most ETH pairs are on `500` or `3000`.

## Slippage and price impact

**Price impact** is the price movement caused by your own trade — larger trades move the price more.

**Slippage** is the difference between the price when you submitted and the price when your transaction is included, caused by other transactions racing ahead in the same block.

Always set `amountOutMinimum` to protect against both:

```python
# Accept at most 0.5% slippage
amount_out_minimum = expected_output * 0.995
```

<Warning>
Setting `amountOutMinimum = 0` is dangerous. An MEV bot can sandwich your transaction — buying before you and selling after — draining nearly all value from your swap. Always set a non-zero minimum output.
</Warning>

## The approve-then-swap pattern

ERC-20 tokens cannot be sent directly to a DEX router as part of the call. You must first **approve** the router to spend your tokens, then call the swap function. This is a two-transaction flow.

**Step 1 — Approve the router:**

```python
from web3 import Web3
from eth_abi import encode

router_address = "0xSwapRouter02Address..."
usdc_address = "0xUSDCAddress..."
amount_in = 1000 * 10**6  # 1000 USDC (6 decimals)

selector = Web3.keccak(text="approve(address,uint256)")[:4].hex()
approve_calldata = "0x" + selector + encode(
    ["address", "uint256"], [router_address, amount_in]
).hex()

await client.call_contract(
    wallet_id=wallet_id,
    chain_id="BASE_ETH",
    contract_addr=usdc_address,
    calldata=approve_calldata,
)
```

**Step 2 — Execute the swap:**

```python
from eth_abi import encode

selector = Web3.keccak(
    text="exactInputSingle((address,address,uint24,address,uint256,uint256,uint160))"
)[:4].hex()

params = encode(
    ["(address,address,uint24,address,uint256,uint256,uint160)"],
    [[
        usdc_address,      # tokenIn
        weth_address,      # tokenOut
        500,               # fee (0.05%)
        wallet_address,    # recipient
        amount_in,         # amountIn
        amount_out_min,    # amountOutMinimum
        0,                 # sqrtPriceLimitX96 (no limit)
    ]]
).hex()

swap_calldata = "0x" + selector + params
result = await client.call_contract(
    wallet_id=wallet_id,
    chain_id="BASE_ETH",
    contract_addr=router_address,
    calldata=swap_calldata,
)
```

See [ERC-20 approvals](/learn/contracts/erc20-approvals) and [ABIs and calldata](/learn/contracts/abi-calldata) for encoding details.

## Pact policy for a swap agent

Your Pact must allow both the ERC-20 approve call and the swap call:

```json
{
  "permissions": ["write:contract_call", "read:wallet"],
  "policies": [
    {
      "name": "allow-usdc-approve-for-router",
      "type": "contract_call",
      "rules": {
        "effect": "allow",
        "when": {
          "chain_in": ["BASE_ETH"],
          "target_in": [{"chain_id": "BASE_ETH", "contract_addr": "0xUSDC..."}],
          "function_in": ["approve(address,uint256)"]
        }
      }
    },
    {
      "name": "allow-uniswap-swap",
      "type": "contract_call",
      "rules": {
        "effect": "allow",
        "when": {
          "chain_in": ["BASE_ETH"],
          "target_in": [{"chain_id": "BASE_ETH", "contract_addr": "0xSwapRouter02..."}],
          "function_in": [
            "exactInputSingle((address,address,uint24,address,uint256,uint256,uint160))"
          ]
        },
        "review_if": {
          "param_gt": {"slippage_bps": "50"}
        }
      }
    }
  ]
}
```

## Multi-hop swaps

When no direct pool exists for a pair, you can route through intermediate tokens using `exactInput`:

```solidity
function exactInput(ExactInputParams calldata params)
    external
    returns (uint256 amountOut);
```

The `path` parameter encodes the route as packed bytes: `tokenA + fee + tokenB + fee + tokenC`. For example, USDC → WETH → ARB routes through two pools.
