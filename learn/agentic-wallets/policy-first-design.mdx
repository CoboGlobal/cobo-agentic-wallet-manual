## The trust fallacy

The intuitive way to think about a safe AI agent is: "use a better model." More capable, more aligned, less likely to make mistakes.

This reasoning is insufficient for anything involving real money. Models hallucinate. Contexts get corrupted. External inputs — prices, contract data, user messages — can be adversarial. Even a perfectly aligned model operating on incorrect information will make incorrect decisions.

The correct framing is: **design for failure, not for perfection**.

A well-designed agentic wallet behaves safely even when the agent makes bad decisions, because the policy layer independently validates every action before it executes.

## Allowlists over blocklists

Security policy can be structured in two ways:

**Blocklist approach**: allow everything except known-bad actions. Easy to start, hard to maintain. Every new attack surface must be explicitly added to the blocklist. A novel attack vector — a new contract, a new function, an unexpected parameter value — passes through unchallenged.

**Allowlist approach**: deny everything except explicitly permitted actions. Hard to start (you must enumerate what you want), but complete. Anything not on the list is blocked by default, including attacks you haven't thought of yet.

For agent wallets, **always use allowlists**:

```json
{
  "name": "allow-specific-swap",
  "type": "contract_call",
  "rules": {
    "effect": "allow",
    "when": {
      "chain_in": ["BASE_ETH"],
      "target_in": [{"chain_id": "BASE_ETH", "contract_addr": "0xUniswapRouter..."}],
      "function_in": ["exactInputSingle((address,address,uint24,address,uint256,uint256,uint160))"]
    }
  }
}
```

This policy allows exactly one function on exactly one contract on exactly one chain. The agent cannot call Aave, cannot call an unknown contract, cannot call `exactOutput` instead of `exactInputSingle`, and cannot operate on Polygon — even if the agent's reasoning concludes that it should.

## Four dimensions of constraint

Well-designed agent policies constrain across four dimensions:

**1. Contract (where)** — which contract addresses can the agent interact with? Use `target_in` to create an explicit allowlist. If you don't recognize a contract address, it should not be in the list.

**2. Function (what)** — which functions on those contracts can the agent call? Use `function_in`. A contract with 20 functions should only expose the 2–3 your agent actually needs.

**3. Parameter (how)** — what values are acceptable within a function call? Use `review_if` and `deny_if` to enforce parameter constraints:

```json
"review_if": {"param_gt": {"slippage_bps": "50"}},
"deny_if": {"param_gt": {"slippage_bps": "200"}}
```

**4. Amount and rate (how much, how often)** — how much value can the agent move? Per transaction, per day, per Pact duration. This is often the most important constraint for limiting blast radius.

## The principle of least privilege

**Least privilege** means granting only the minimum access necessary to accomplish the task — nothing more.

Applied to agent wallets:

| Instead of... | Do this |
|---|---|
| Allow all contracts | Allowlist only the specific contracts this agent needs |
| Allow all functions | Allowlist only the specific functions this agent calls |
| Set a large amount cap "just in case" | Set the cap to the realistic maximum this task needs |
| Grant a 30-day Pact for a one-time task | Set a duration that matches the actual task window |
| Allow both testnets and mainnets | Restrict to only the chain this agent operates on |

Violations of least privilege feel harmless until they are not. An agent authorized for "any contract, any function" on mainnet is not meaningfully safer than one with a private key.

## Progressive trust

You do not need to get the policy right on day one. Start narrow and expand as you observe behavior:

```
Week 1:  allow 1 contract, 1 function, $100/day cap, 7-day Pact
         → Review audit logs daily. No surprises?

Week 2:  expand to 2 contracts, $500/day cap, 14-day Pact
         → Review audit logs weekly. Still clean?

Week 4:  add 2 more functions, $2000/day cap, 30-day Pact
```

This approach lets you verify agent behavior in a low-stakes environment before trusting it with larger scope. Each Pact renewal is an opportunity to review what the agent actually did vs. what you authorized.

## Designing the review threshold

Not every out-of-bounds action needs to be an outright denial. The `review_if` clause creates a **manual approval gate** for edge cases:

```json
"review_if": {"param_gt": {"amount_usd": "1000"}}
```

This means: auto-approve transactions under $1000, but require owner approval above that. The agent is not blocked — it queues the transaction and notifies the owner. This is useful when:

- The agent occasionally needs to exceed normal limits for legitimate reasons.
- You want visibility into larger operations without requiring approval for routine ones.
- You are calibrating limits and don't yet know the right threshold.

Use `deny_if` for hard limits you never want exceeded regardless of context:

```json
"deny_if": {"param_gt": {"slippage_bps": "100"}}
```

## Policy as documentation

A well-written Pact is a human-readable specification of what the agent is supposed to do. If someone reads the policy and cannot tell what the agent's job is, the policy is probably too broad.

Good policy names communicate intent:
- `allow-usdc-to-eth-swap-on-base` — clear
- `allow-contract-call-1` — opaque, likely too broad

The agent's `intent` field in the Pact request is also part of this documentation layer — the owner should be able to read the intent, look at the policy rules, and verify that the rules match the stated intent.
