## What is a testnet

A **testnet** is a separate blockchain network that mimics a mainnet but uses tokens with no real-world value. Testnets let you:

- Test agent logic without risking real funds
- Simulate DeFi interactions before going live
- Experiment with contract calls and policy configurations
- Iterate quickly without worrying about transaction costs

Testnet tokens are freely available from **faucets** — services that dispense small amounts on request.

## Cobo testnet support

| Cobo chain ID | Testnet | Mirrors |
|---|---|---|
| `SETH` | Ethereum Sepolia | `ETH` (Ethereum mainnet) |
| `TBASE_SETH` | Base Sepolia | `BASE_ETH` (Base mainnet) |
| `SOLDEV_SOL` | Solana Devnet | `SOL` (Solana mainnet) |

New wallets created via onboarding default to Base Sepolia (`TBASE_SETH`). All quickstart guides and recipes use testnet by default.

## Getting testnet tokens

Cobo provides a built-in faucet for testnet tokens:

```bash
# Request Sepolia ETH
caw faucet --chain SETH --token SETH --amount 0.1

# Request Sepolia USDC
caw faucet --chain SETH --token SETH_USDC --amount 100

# Request Base Sepolia ETH
caw faucet --chain TBASE_SETH --token TBASE_SETH --amount 0.1

# Request Solana Devnet SOL
caw faucet --chain SOLDEV_SOL --token SOLDEV_SOL --amount 1
```

Or via the API: [Request faucet tokens](/api-reference/faucet/request-tokens).

<Info>
Faucet requests are rate-limited. For high-volume testing, consider using the protocol's own public testnet faucets in addition to the Cobo faucet.
</Info>

## Testnet contract addresses

Most protocols deploy separate contracts on each testnet. Uniswap, Aave, and other DeFi protocols have dedicated Sepolia and Base Sepolia deployments at different addresses than mainnet.

Always look up testnet addresses from the protocol's official documentation before writing policies or calldata. For example:
- Uniswap v3 deployments: [docs.uniswap.org/contracts/v3/reference/deployments](https://docs.uniswap.org/contracts/v3/reference/deployments)
- Aave v3 deployments: [docs.aave.com/developers/deployed-contracts](https://docs.aave.com/developers/deployed-contracts)

<Warning>
Mainnet contract addresses do not exist on testnets. If your agent submits a contract call to a mainnet address on a testnet, the transaction will fail with a "not a contract" error.
</Warning>

## Moving to mainnet

When you are ready to go live:

1. Open the Web Console or Human App and navigate to your wallet.
2. Select **Add Mainnet** and follow the AI assistant conversation.
3. The assistant guides you through confirming your policy settings before any mainnet access is enabled.

This step cannot be skipped — it is a deliberate safety gate requiring explicit owner confirmation before real funds can be handled.

Testnet and mainnet addresses share the same wallet container: your wallet ID stays the same, and existing Pact authorizations and policies continue to apply.

## Testnet limitations

Testnets can behave differently from mainnet:

- **Liquidity**: Testnet DeFi pools are shallower — large swaps may fail or produce extreme slippage.
- **Protocol availability**: Not all protocols are deployed on all testnets.
- **Stability**: Testnets occasionally reset or experience outages.
- **Block time**: Usually similar to mainnet but can vary.

Test the core logic and policy enforcement on testnet; then verify again on mainnet with small amounts before running agents at full scale.
