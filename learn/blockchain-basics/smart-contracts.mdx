## What is a smart contract

A **smart contract** is a program deployed on a blockchain at a specific address. Like a wallet address, a contract address is a 42-character hex string on EVM chains. Unlike a wallet, it has code attached to it.

When you send a transaction to a contract address with encoded function data, the contract executes that function. The execution is deterministic — given the same inputs and blockchain state, every node on the network reaches the same result. Once executed, the result is permanent.

Contracts power all of DeFi: Uniswap's swap logic lives in contracts, Aave's lending pool is a contract, and every ERC-20 token is a contract.

## Read functions vs. write functions

| Type | Gas required | Changes state | Examples |
|---|---|---|---|
| **Read (view/pure)** | No | No | `balanceOf()`, `getReserves()`, `getUserAccountData()` |
| **Write** | Yes | Yes | `transfer()`, `swap()`, `deposit()`, `approve()` |

Read calls are free and instant — they query the current chain state without submitting a transaction. You can call them directly against any public RPC endpoint.

Write calls submit a signed transaction and cost gas. With Cobo Agentic Wallet, all write calls go through the policy engine before being signed and broadcast.

## What is an ABI

An **ABI** (Application Binary Interface) is a JSON document describing a contract's functions, their parameter types, and return types. It is the schema that tells your code how to encode calls to the contract and decode responses.

Example ABI entry for ERC-20 `transfer`:

```json
{
  "name": "transfer",
  "type": "function",
  "inputs": [
    {"name": "to", "type": "address"},
    {"name": "amount", "type": "uint256"}
  ],
  "outputs": [{"name": "", "type": "bool"}],
  "stateMutability": "nonpayable"
}
```

Without the ABI, you would need to manually encode raw bytes. With the ABI, libraries like `eth_abi` (Python), `viem` (TypeScript), or `web3.py` handle encoding for you. See [ABIs and calldata](/learn/contracts/abi-calldata) for how encoding works.

## Events

Contracts emit **events** when significant state changes occur. Events are stored in the transaction receipt and indexed by nodes.

```solidity
// ERC-20 Transfer event
event Transfer(address indexed from, address indexed to, uint256 value);

// Uniswap v3 Swap event
event Swap(
    address indexed sender,
    address indexed recipient,
    int256 amount0,
    int256 amount1,
    ...
);
```

Off-chain services — including Cobo's webhook infrastructure — can listen for events to react to on-chain activity. Block explorers display events in the transaction receipt.

## Calling contracts with Cobo

To call a write function on a contract, you need:

1. The contract address
2. The ABI-encoded calldata for the function call
3. The chain ID

```bash
caw tx call <WALLET_ID> \
  --contract 0xContractAddress \
  --calldata 0xEncodedFunctionData \
  --chain BASE_ETH
```

```python
result = await client.call_contract(
    wallet_id=wallet_id,
    chain_id="BASE_ETH",
    contract_addr="0xContractAddress",
    calldata="0xEncodedFunctionData",
)
```

The policy engine checks the contract address and function selector against your Pact's `allowed_contracts` and `function_in` rules before signing.

See [ABIs and calldata](/learn/contracts/abi-calldata) for how to generate the `calldata` value.

## Finding contract addresses

- **Block explorers**: Search for the protocol name on Etherscan, Basescan, or Polygonscan. Verified contracts show a green checkmark and their full source code.
- **Protocol documentation**: Uniswap, Aave, Curve, and other major protocols maintain official deployment address lists.
- **Cobo API**: [List supported assets](/api-reference/metadata/list-assets) returns token contract addresses for all tokens Cobo supports.

<Warning>
Contract addresses are network-specific. The Uniswap router on Base mainnet (`BASE_ETH`) is a different address than on Base Sepolia (`TBASE_SETH`). Always verify you are using the address for the correct network.
</Warning>

## Contract upgrades and proxies

Many DeFi protocols use **proxy patterns** — the contract address you interact with is a proxy that delegates logic to an implementation contract. This allows the protocol to upgrade its logic without changing the address you use.

For policy purposes, use the proxy address in `target_in` rules — this is the address your agent actually sends transactions to.
