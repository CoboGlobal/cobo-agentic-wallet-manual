---
title: "Key Share Recovery"
description: "How to restore a lost or inaccessible MPC key share without exposing or reconstructing the private key."
---

Cobo Agentic Wallet uses MPC threshold signing to ensure that no single party holds a complete private key. This same architecture provides a recovery path when a key share is lost — without ever revealing or reconstructing the underlying private key.

This page explains why recovery is needed, how the resharing protocol works, and what to expect in each failure scenario.

<Info>
For background on what key shares are and why there are multiple shares per wallet, see [MPC Key Shares](/products/agentic-wallet/manual/security/mpc-key-shares).
</Info>

## Why recovery is needed

A key share is a piece of cryptographic material stored on a specific device or infrastructure node. Unlike passwords or seed phrases, it is not something you can simply memorize and retype — it must be present on the signing device at the moment a transaction is being signed.

Events that can make a key share inaccessible include:

- **Agent node replacement**: the machine running the agent's TSS node is decommissioned or reprovisioned, and the profile directory (`~/.cobo-agentic-wallet/profiles/`) was not backed up.
- **Lost phone (Human App)**: the mobile device holding the user's on-device share is lost, stolen, or reset.
- **Hardware failure**: storage failure or OS reinstall on the agent's machine destroys the local share files.
- **Infrastructure migration**: the agent service is moved to new infrastructure without transferring the profile directory.

In all of these cases, the wallet's funds are not lost. The blockchain address and the public key still exist on-chain. The wallet simply cannot sign new transactions until a new share is generated for the affected party.

## The resharing protocol

Key share recovery uses a cryptographic **resharing** (or re-sharing) protocol. Resharing allows an existing quorum to generate new key shares for a new set of parties — without ever combining the existing shares into a complete private key.

The resharing protocol has the following properties:

- The existing quorum cooperates to produce new shares for a new target configuration.
- The full private key is never reconstructed or transmitted at any point.
- The wallet's public key (and therefore its blockchain address) remains the same — no on-chain migration is needed.
- After resharing completes, the old shares are cryptographically invalidated. Any party holding an old share can no longer contribute to a signing quorum.

<Warning>
Resharing invalidates old shares. If you hold an old share after a reshard, it cannot be used to sign or to perform a second reshard. Complete the resharing process before decommissioning the original infrastructure.
</Warning>

## Scenario 1: Agent node replaced (agent-managed, 2/2)

In the agent-managed configuration, the two shares are held by Cobo (HSM) and the agent service node.

If the agent's share is lost:

<Steps>
  <Step title="Contact Cobo support">
    Open a support ticket or contact your Cobo account manager. You will need to verify ownership of the wallet.
  </Step>
  <Step title="Provide new node details">
    Provide the identity credentials of the new agent node. This is used to bind the new share to the correct principal.
  </Step>
  <Step title="Resharing ceremony">
    Cobo initiates a resharing ceremony using its existing share (the only available share at this point, since the agent's share is lost). The new share is generated for the replacement node.
  </Step>
  <Step title="Verify and resume">
    Once resharing completes, verify that the new node can participate in a test signing operation. The old share, if it still exists anywhere, is now invalidated.
  </Step>
</Steps>

<Note>
For 2/2 wallets, the agent's share is the only party-controlled share. Without Cobo's cooperation, resharing cannot be initiated. This is by design — it prevents unauthorized resharing by a compromised agent.
</Note>

## Scenario 2: Human App device lost (Human App, 2/3)

In the Human App configuration, three shares exist: Cobo (HSM), the agent service node, and the user's mobile device. Any two of the three can sign.

If the mobile device is lost:

<Steps>
  <Step title="The remaining quorum is sufficient">
    Because this is a 2/3 scheme, the remaining two shares (Cobo + agent) form a valid quorum. New transactions can still be signed autonomously — no immediate access outage occurs.
  </Step>
  <Step title="Initiate resharing from the Human App">
    On a new device, open the Human App and follow the account recovery flow. The app will initiate a resharing ceremony using the existing 2/3 quorum.
  </Step>
  <Step title="New device share generated">
    The resharing protocol produces a new device share on the replacement device. The old device's share is invalidated.
  </Step>
  <Step title="Restore Human App approvals">
    Once resharing is complete, the user can again co-sign transactions and approve pending operations from the new device.
  </Step>
</Steps>

<Info>
Human App recovery is self-service. You do not need to contact Cobo support if you have access to any two of the three original shares — or if you still have your Cobo account credentials to authenticate the resharing request.
</Info>

## Scenario 3: Agent-managed wallet — total agent loss

If the agent's profile directory is fully destroyed and no backup exists:

<Steps>
  <Step title="Contact Cobo support">
    Cobo support initiates a reshard using Cobo's HSM share — the only remaining valid share. Identity verification is required.
  </Step>
  <Step title="Provision a new agent node">
    Set up a new agent node and run `caw onboard` to register the new node's identity. Do not generate a new wallet — recovery is linking the existing wallet to a new node.
  </Step>
  <Step title="Resharing to the new node">
    Once the new node identity is registered, Cobo completes the resharing ceremony and the new node receives its share.
  </Step>
  <Step title="Resume operations">
    The new node can now co-sign transactions. Back up the profile directory immediately after recovery to prevent recurrence.
  </Step>
</Steps>

## What resharing does not do

It is important to be clear about what the resharing protocol does not do:

- **It does not reveal the private key.** At no point during resharing does the full private key exist in one place.
- **It does not change the wallet address.** The blockchain address is derived from the public key, which remains unchanged.
- **It does not move or migrate funds.** Your funds stay in the same wallet. Only the signing parties change.
- **It does not give Cobo unilateral control.** Even during a Cobo-initiated resharing, the process is bound to the identity credentials of the wallet owner. Cobo cannot reship to an arbitrary new party.

## After resharing: old shares are invalidated

Once a resharing ceremony completes successfully, the previous shares are cryptographically invalidated. This is an automatic property of the TSS resharing protocol — there is no separate "revoke old share" step required.

If an attacker had previously obtained a copy of an old share, that copy becomes useless after resharing. It cannot be used to contribute to a signing quorum for the current share configuration.

## Prevention: backing up agent key shares

The most reliable way to avoid needing a recovery is to back up the agent's profile directory immediately after onboarding:

```
~/.cobo-agentic-wallet/profiles/
```

Store backups in secure, access-controlled storage. If the directory is restored from backup, the TSS node will resume using the original share without requiring a resharing ceremony.
