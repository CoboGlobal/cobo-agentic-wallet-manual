## What is a transaction

A **transaction** is a signed message submitted to a blockchain that changes state — transferring tokens, calling a contract function, or deploying a new contract. Once included in a block, transactions are permanent and irreversible.

Every EVM transaction has:

| Field | Description |
|---|---|
| `from` | The account initiating and paying gas |
| `to` | The recipient address or contract |
| `value` | Amount of native token to send (can be zero) |
| `data` | Encoded function call for contracts; empty for plain transfers |
| `gas limit` | Maximum computation the sender is willing to pay for |
| `nonce` | A counter that prevents duplicate transactions from being replayed |

## Transaction hash

Every submitted transaction receives a unique **transaction hash** (tx hash) — a 32-byte hex string like `0x4a3f8b2c...`. This is how you look up a transaction on a block explorer:

- Ethereum / Sepolia: [etherscan.io](https://etherscan.io) / [sepolia.etherscan.io](https://sepolia.etherscan.io)
- Base / Base Sepolia: [basescan.org](https://basescan.org) / [sepolia.basescan.org](https://sepolia.basescan.org)
- Polygon: [polygonscan.com](https://polygonscan.com)

Cobo returns the transaction hash after successful submission.

## Gas fees

**Gas** is the unit of computational work on EVM chains. Every operation (storing a value, running code, transferring tokens) costs a fixed amount of gas. The total fee is:

```
Transaction fee = gas used × gas price (in Gwei)
```

Gas is paid in the chain's native token (ETH on Ethereum and Base, MATIC on Polygon). On Solana, fees are paid in lamports (SOL).

Gas prices fluctuate with network congestion. During high activity, fees can spike significantly.

## Cobo Gasless (Fee Station)

Cobo Agentic Wallet includes **Cobo Gasless**, which automatically sponsors gas fees on behalf of your wallet. You do not need to hold ETH, MATIC, SOL, or any other native token to execute transactions.

This means:
- An agent wallet holding only USDC can still transfer USDC.
- Token swaps and contract calls work without pre-funding gas.
- Developers do not need to build gas management into their agents.

Gas sponsorship is applied automatically at the signing layer — no configuration required.

## Checking transaction status

After submitting a transaction, poll for its confirmation status:

```bash
caw tx get <WALLET_ID> <TX_ID>
```

Or via the Python SDK:

```python
tx = await client.get_transaction(wallet_id=wallet_id, tx_id=tx_id)
print(tx["status"])  # pending, confirmed, failed
```

## Transaction lifecycle

```
Agent submits → Policy engine validates → Cobo signs → Broadcast to network
    → Included in block → Additional confirmations → Confirmed
```

The policy engine step is synchronous — if the transaction violates a policy, it is blocked before reaching the network. If it is within policy but requires approval, it is queued as a pending operation.

## Transaction finality

Transactions are not final the moment they are submitted. They must be included in a block and confirmed by subsequent blocks.

| Chain | Time to first confirmation |
|---|---|
| Ethereum mainnet | ~12 seconds |
| Base mainnet | ~2 seconds |
| Polygon mainnet | ~2 seconds |
| Solana mainnet | ~400 milliseconds |
| Ethereum Sepolia | ~12 seconds |
| Base Sepolia | ~2 seconds |

Cobo waits for block inclusion before marking a transaction as confirmed.

## Failed transactions

A transaction can fail at two stages:

**Before submission** — blocked by the Cobo policy engine (violates a policy rule). No gas is consumed. The agent receives a `deny` or `require_approval` response.

**After submission** — the transaction is broadcast but reverts on-chain (e.g., insufficient token balance, slippage exceeded, contract revert). Gas is consumed up to the point of failure. Cobo reports the transaction as `failed` with the revert reason if available.
