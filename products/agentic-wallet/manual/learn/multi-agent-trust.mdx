## Why multi-agent architectures emerge

Complex tasks often exceed what a single agent can handle efficiently. A trading system might split into:

- An **orchestrator** that decides the overall strategy.
- A **market data agent** that fetches prices and signals.
- An **execution agent** that submits the actual transactions.
- A **risk monitor** that checks portfolio health and alerts the orchestrator.

Each specialist agent does one thing well. The orchestrator coordinates them.

This pattern is powerful, but it introduces a trust question: **when one agent delegates work to another, what authority does the subagent inherit?**

## The core principle: authority does not flow automatically

In multi-agent systems, authority must be **explicitly scoped** — not implicitly inherited.

Just because an orchestrator has a [Pact](/products/agentic-wallet/manual/security/pact-mechanism) that allows contract calls does not mean a subagent it spins up also has that authority. Each agent must operate under its own explicitly scoped credentials.

The correct model:

```
Owner approves Pact for orchestrator
    └── Orchestrator receives scoped API key (can call contracts A, B, C)
            └── Orchestrator creates a sub-Pact for execution agent
                    └── Owner approves sub-Pact (narrower: only contract A)
                            └── Execution agent receives its own scoped API key
```

The execution agent's key is more restrictive than the orchestrator's. It can only do what its own Pact allows — not everything the orchestrator is allowed to do.

## Why this matters: containing subagent compromise

If a subagent is compromised — through prompt injection, a bug in its code, or a supply chain attack on a library it uses — the damage is bounded by that subagent's Pact scope, not the orchestrator's.

**Without scoped subagents:**
- Orchestrator Pact: can call any function on Uniswap, Aave, and Curve; move up to $50,000/day.
- Execution agent uses orchestrator's key.
- Compromised execution agent: can move $50,000/day to any allowed contract.

**With scoped subagents:**
- Orchestrator Pact: full scope.
- Execution agent Pact: only `exactInputSingle` on Uniswap; max $1,000/transaction.
- Compromised execution agent: can execute Uniswap swaps within the $1,000 cap. Cannot touch Aave, Curve, or larger amounts.

This is the principle of least privilege applied to multi-agent systems.

## Patterns for multi-agent authorization

### Pattern 1: Parallel specialists

Each specialist agent has its own Pact, independently approved by the owner. The orchestrator coordinates them but does not hold credentials on their behalf.

```
Owner
├── approves Pact A → Market Data Agent (read: prices, balances)
├── approves Pact B → Execution Agent (write: swaps, max $1,000)
└── approves Pact C → Risk Monitor (read: positions, no write)

Orchestrator coordinates agents A, B, C — does not hold their keys
```

This is the simplest and most auditable pattern. Each agent's activity appears separately in audit logs.

### Pattern 2: Orchestrator with delegated sub-Pacts

The orchestrator holds a Pact and can create narrower sub-Pacts for subagents. The owner approves the orchestrator's Pact; subagent Pacts are approved within the orchestrator's scope.

```
Owner
└── approves Pact for orchestrator (write: swaps and deposits, max $10,000)

Orchestrator
├── creates sub-Pact for DCA agent (write: swaps only, max $500/day)
└── creates sub-Pact for yield agent (write: deposits only, max $5,000)
```

Subagents cannot exceed the orchestrator's scope — the policy engine enforces this automatically.

### Pattern 3: Sequential pipeline

Each agent in a chain receives only the authority needed for its step and passes results (not credentials) to the next agent.

```
Signal Agent → produces trade signal (no wallet access)
    ↓ (signal data, no keys)
Risk Agent → validates signal against portfolio rules (read-only)
    ↓ (go/no-go decision, no keys)
Execution Agent → submits transaction (write, scoped to approved contracts)
```

Agents that do not need wallet write access should not have it. Read-only agents (market data, monitoring) can operate without any Pact by calling public RPC endpoints directly.

## Trust levels between agents

When designing a multi-agent system, assign each agent a trust tier:

| Trust tier | What this agent can do | Example roles |
|---|---|---|
| **Untrusted** | No wallet access; only reads public data | Price feeds, news scrapers |
| **Read-only** | Can query wallet state; no transactions | Portfolio monitors, reporting agents |
| **Constrained write** | Can submit transactions within narrow scope | Execution agents, DCA bots |
| **Broad write** | Wide transaction scope; carefully supervised | Strategy orchestrators |

Never elevate an agent's trust tier without corresponding policy constraints. An orchestrator that receives broad scope should not propagate that scope to subagents — scope should narrow at each delegation step.

## Agent-to-agent communication

When agents communicate (passing signals, strategies, or instructions between them), apply the same prompt injection caution as with external data:

- **Structure over text**: pass structured data (JSON objects with typed fields) between agents, not natural language instructions. A compromised upstream agent cannot inject text instructions into a downstream agent's structured input.
- **Validate received data**: the receiving agent should validate that the data it received matches expected types and ranges before acting on it.
- **Do not relay credentials**: agents should never pass API keys, Pact IDs, or authentication tokens to other agents. Each agent authenticates independently.
