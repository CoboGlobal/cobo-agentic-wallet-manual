## Base URL

```
https://api.sandbox.agenticwallet.cobo.com
```

All versioned endpoints are prefixed with `/api/v1`. For example:

```
https://api.sandbox.agenticwallet.cobo.com/api/v1/wallets
```

## Authentication

Every request must include an `X-API-Key` header:

```bash
curl -H "X-API-Key: <your-api-key>" \
  https://api.sandbox.agenticwallet.cobo.com/api/v1/wallets
```

You obtain an API key by completing the [onboarding flow](/products/agentic-wallet/manual/get-started/connect-agent-human-app). The key is bound to your account and wallet.

### API key scopes

Each API key carries a set of scopes that control which operations it can perform. A scope has the form `resource:action`, where `action` is either `read` or `write`.

Examples:
- `wallets:read` — read wallet data
- `wallets:write` — create and modify wallets
- `transactions:write` — submit transfers and contract calls
- `*` — full access to all resources

GET requests require a `read` scope; POST, PATCH, and DELETE requests require a `write` scope. The [Dry-run policy](/products/agentic-wallet/manual/reference/dry-run-policy) operation is an exception — it requires `policies:read` despite being a POST.

## Response format

All successful responses use a standard envelope:

```json
{
  "success": true,
  "result": { ... },
  "meta": {
    "total": 42,
    "offset": 0,
    "limit": 50
  }
}
```

- `result` contains the operation's return value — an object for single-resource operations, an array for list operations.
- `meta` is present on list responses and contains pagination information.

The Python SDK automatically unwraps `result` — method return values are the unwrapped data, not the full envelope.

## Pagination

Most list operations use **offset-based** pagination:

| Parameter | Default | Description |
|---|---|---|
| `limit` | `50` | Maximum items to return. Range: [1, 200]. |
| `offset` | `0` | Number of items to skip before returning results. |

The response `meta` field contains `total`, `offset`, and `limit`.

[List audit logs](/products/agentic-wallet/manual/reference/list-audit-logs) uses **cursor-based** pagination instead:

| Parameter | Default | Description |
|---|---|---|
| `limit` | `50` | Maximum items to return. Range: [1, 200]. |
| `cursor` | — | Pass the `next_cursor` value from the previous response to retrieve the next page. |

## Error responses

Errors follow one of two shapes depending on the cause.

**Authentication and authorization errors (401, 403)**

```json
{
  "detail": "Insufficient API key scope. Required: 'wallets:read'"
}
```

**Policy denial errors (403)**

When a transaction is blocked by an active policy, the `detail` field contains a structured denial object:

```json
{
  "detail": {
    "code": "DAILY_VALUE_LIMIT_EXCEEDED",
    "reason": "daily_value_limit_exceeded",
    "details": {
      "window": "daily",
      "limit_value": "1000",
      "requested": "1500",
      "remaining": "0",
      "unit": "USD",
      "resets_at": "2026-04-01T00:00:00Z"
    },
    "suggestion": "Retry with USD <= 0 or retry after 2026-04-01T00:00:00Z when the daily value window resets."
  }
}
```

The `code` field is machine-readable. Common values: `TRANSFER_LIMIT_EXCEEDED`, `DAILY_VALUE_LIMIT_EXCEEDED`, `CHAIN_RESTRICTED`, `CONTRACT_NOT_WHITELISTED`, `WALLET_FROZEN`, `APPROVAL_REQUIRED`.

**Validation errors (422)**

```json
{
  "detail": [
    {
      "loc": ["body", "amount"],
      "msg": "field required",
      "type": "missing"
    }
  ]
}
```

## HTTP status codes

| Status | Meaning |
|---|---|
| `200` | Success |
| `201` | Resource created |
| `400` | Bad request (for example, insufficient balance) |
| `401` | Missing or invalid API key |
| `403` | Forbidden — insufficient scope or policy denial |
| `404` | Resource not found |
| `422` | Request body or query parameter validation failed |
| `502` | Upstream Cobo WaaS API error |
| `5xx` | Server error |

**Python SDK exceptions**

| Exception | Raised on |
|---|---|
| `AuthenticationError` | 401 |
| `PolicyDeniedError` | 403 policy denial — carries a `.denial` attribute with the structured denial fields |
| `NotFoundError` | 404 |
| `ServerError` | 5xx |

For generic 403 errors (for example, wrong scope), the SDK raises the base `APIError`.

## Chain and token IDs

All operations that accept a `chain_id` or `token_id` use Cobo-specific identifiers — not chain names or common ticker symbols.

Examples: `SETH` (Ethereum Sepolia), `BASE_ETH` (Base mainnet), `SOL` (Solana mainnet), `SETH_USDC` (USDC on Ethereum Sepolia).

Call [List supported chains](/products/agentic-wallet/manual/reference/list-chains) or [List supported assets](/products/agentic-wallet/manual/reference/list-assets) to retrieve all IDs supported by the service.

## Health endpoints

Two endpoints are available for monitoring — neither requires authentication:

| Endpoint | Description |
|---|---|
| `GET /health` | Returns service health and uptime status |
| `GET /api/v1/ping` | Lightweight liveness check |
